# 后端学习计划 — 前端开发者视角

> 目标：**短期能独立开发 CRUD，长期系统掌握后端技术栈。**
> 不用全部学完再动手，先拿下 CRUD 能力就能参与业务开发。

## 🎯 短期目标：独立开发 CRUD（2-3 周）

完成以下任务即视为 CRUD 目标达成 —— 这是后端开发的"驾照"。

**用户管理 API：**

| 接口 | 说明 |
|------|------|
| `POST /users` | 创建用户 |
| `GET /users/{id}` | 查询用户详情 |
| `GET /users` | 分页列表 |
| `PUT /users/{id}` | 更新用户 |
| `DELETE /users/{id}` | 删除用户 |

---

## P0：CRUD 学习路径（预计 2-3 周）

### 步骤一：环境搭建 + 第一条 API（2-3 天）

**必须掌握：**
| 知识点 | 要求 | 说明 |
|--------|------|------|
| JDK 21 安装 / IDEA 配置 | 能创建 Spring Boot 项目 | 用 [start.spring.io](https://start.spring.io) 生成项目 |
| Maven 基础 | 看懂 `pom.xml`，会 `mvn clean package` | 不深究生命周期，先会用 |
| Spring Boot 启动类 + `@SpringBootApplication` | 知道项目从哪启动 | |
| `@RestController` + `@GetMapping`/`@PostMapping` | 写一条 GET `/hello` 和 POST `/echo` | **第一个里程碑** |

**可跳过：** Maven 高级插件、多模块构建、自动配置原理

### 步骤二：分层架构 + 数据库 CRUD（3-5 天）

这是后端开发的**核心骨架**。

**必须掌握：**
| 知识点 | 要求 |
|--------|------|
| Controller → Service → Repository 三层分层 | 理解各层职责，这是 DDD 的简化版 |
| Spring Data JPA + `@Entity` | 写一个实体类映射到表 |
| JPA Repository 基本 CRUD（`save`, `findById`, `findAll`, `deleteById`） | 能完成增删改查 |
| `@Service` + 构造器注入 | 理解依赖注入 |
| `application.yml` 配置 MySQL 数据源 | 能连上 MySQL 8.0 |
| `@RequestBody` / `@RequestParam` / `@PathVariable` | 参数接收 |
| 统一返回结果包装 `R<T>` | 跟团队风格一致 |

**练手：** 实现用户管理全部 5 个 API（见顶部表格）

**可跳过：** JPA 复杂关联映射、事务传播行为、连接池调优

### 步骤三：完善与巩固（2-3 天）

**必须掌握：**
| 知识点 | 要求 |
|--------|------|
| `@Valid` + Bean Validation | 请求参数校验 |
| `@RestControllerAdvice` + `@ExceptionHandler` | 统一异常处理 |
| 分页查询 `Pageable` + `Page<T>` | 列表分页 |
| 日志 SLF4J | 基本日志输出 |

> ✅ **P0 完成标志**：能从头搭建项目 → 建表 → 写 CRUD → 加校验 → 统一异常处理，不查教程独立完成。

---

## ⭐ P1：增强能力（后续学习，预计 4-6 周）

P0 完成后，按以下顺序拓展。

### 阶段四：DDD 入门 — 理解项目代码结构（1-2 天）

> 团队用 DDD 思想，但你**不需要从头学 Evans 那本蓝皮书**。只需要理解项目中代码包怎么分层。

| 概念 | 对应代码位置 | 等价理解 |
|------|-------------|---------|
| Controller（接口层） | `adapter/` | 接收请求、返回响应 |
| Application Service（应用层） | `application/` | 编排逻辑、事务脚本 |
| Domain（领域层） | `domain/` | 核心业务逻辑、实体、值对象 |
| Infrastructure（基础设施层） | `infrastructure/` | DB、MQ、外部服务 |
| Repository（仓储） | `domain/repository/` | 数据访问抽象（在 DDD 中不等于 DAO） |

**可跳过：** 领域事件、聚合设计、CQRS、Event Sourcing

### 阶段五：Nacos + 配置管理（1-2 天）

| 知识点 | 要求 |
|--------|------|
| Nacos 本地启动（docker-compose） | 跑起来 |
| `bootstrap.yml` 配置 Nacos 地址 | 从 Nacos 拉配置 |
| Nacos 配置中心：新建 Data ID、发布配置 | 在界面上操作一次 |
| `@RefreshScope` + 动态刷新 | 在线改配置不重启 |
| Nacos 服务注册 + `@EnableDiscoveryClient` | 知道项目注册到 Nacos |

**可跳过：** Nacos 集群部署、权限控制、服务发现高级用法

### 阶段六：Redis 缓存（1-2 天）

| 知识点 | 要求 |
|--------|------|
| Redis 基本数据类型（String、Hash） | 知道存什么样子 |
| `spring-boot-starter-data-redis` + `RedisTemplate` | 缓存查询结果 |
| `@Cacheable` / `@CacheEvict` 声明式缓存 | 最常用的方式 |
| 缓存常见问题：缓存穿透、雪崩 | 知道概念和简单处理 |

练手：给用户管理项目的 `GET /users/{id}` 加 `@Cacheable`

**可跳过：** Redis 持久化、集群、哨兵、Lua 脚本、分布式锁

### 阶段七：RabbitMQ — 消息发送 & 消费（1-2 天）

| 知识点 | 要求 |
|--------|------|
| Producer / Consumer / Queue / Exchange / Routing Key | 能画出来这几个的关系 |
| 直连交换机（Direct Exchange） | 点对点消息 |
| 主题交换机（Topic Exchange） | 按规则路由消息 |
| `@RabbitListener` 消费消息 | 写一个消费者 |
| `RabbitTemplate.convertAndSend()` 发送消息 | 写一个生产者 |

练手：用户注册后发送"新用户通知"消息

**可跳过：** 死信队列、延迟队列、消息确认机制、集群

---

## 📚 P2：按需深入学习（后续，预计 5-7 周）

- **MySQL 进阶**：索引优化、执行计划、事务隔离级别、MVCC
- **Redis 深入**：分布式锁（Redisson）、持久化策略
- **Spring Boot 深入**：IoC 原理、AOP、自动配置、测试（JUnit 5 + Mockito + Testcontainers）
- **RabbitMQ 深入**：死信队列、消息可靠性、重试机制
- **Elasticsearch 8.x**：全文检索、Spring Data ES、ELK
- **DDD 战术设计**：聚合、领域事件、仓储模式

---

## 🔮 P3：进阶精通（长期目标，预计 10-16 周）

- **Spring Cloud 微服务**：Gateway、OpenFeign、Sentinel
- **Spring Cloud Alibaba**：Seata 分布式事务、RocketMQ
- **DDD 战略设计**：限界上下文、上下文映射
- **FDS 文件存储**：对象存储接入
- **Talos 流处理**：实时数据 Pipeline

---

## 优先级热力图

```
🎯 P0 立即学            → Spring Boot 分层 + JPA/MySQL CRUD（2-3 周）
⭐ P1 P0 完成后尽快学   → DDD 看懂包结构 + Nacos + Redis + RabbitMQ 基础（4-6 周）
📚 P2 按需学           → MySQL/Redis/RabbitMQ 深入 + ES + DDD 战术（5-7 周）
🔮 P3 后面学           → Spring Cloud 微服务 + Alibaba + FDS + Talos（10-16 周）
```

## 建议的学习节奏

- **先拿驾照**：P0 期间集中全部精力，完成 CRUD 目标再进入 P1
- **每天连续**：建议不要中断超过 2 天
- **学练比 1:1**：看文档半小时就要写代码半小时
- **模仿优先**：先看团队现有代码里类似场景怎么写的

## 需要等项目代码后才能补充的内容

- 项目的 Maven 模块结构
- 现有的包目录组织
- 自定义的统一返回 `R<T>` 实现
- Nacos 上实际的 Data ID 和配置项
- RabbitMQ 交换机 / 队列的命名规范
- 团队约定的异常处理方式
- 现有的 Docker Compose 或部署配置
